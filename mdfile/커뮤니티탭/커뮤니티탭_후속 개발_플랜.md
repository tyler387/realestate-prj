# 커뮤니티탭 후속 개발 플랜

## 1. 문서 목적

이 문서는 커뮤니티탭 기능 점검 결과를 바탕으로 후속 개발 범위, 우선순위, 검증 기준을 정리한다.

현재 커뮤니티탭은 빌드와 백엔드 테스트는 통과하지만, 사용자 화면에 직접 드러나는 한글 인코딩 문제와 권한 정책 불일치, 마이페이지 조회 보안, 검색/페이징 부재, 캐시 갱신 누락 가능성이 남아 있다.

## 2. 현재 상태 요약

### 2.1 정상 동작 확인

1. 프론트엔드 프로덕션 빌드 통과
   - 명령어: `npm run build`
   - 위치: `frontend`

2. 백엔드 테스트 재실행 통과
   - 명령어: `.\gradlew.bat test --rerun-tasks`
   - 위치: `backend`

3. 구현 확인된 기능
   - 전체 커뮤니티 / 우리 아파트 탭 분리
   - 게시판 코드 기반 필터링
   - 게시글 목록, 상세, 작성, 삭제
   - 댓글 작성, 삭제
   - 좋아요 토글
   - 인기 키워드, 인기글, 댓글 많은 글 사이드 영역
   - 집계 테이블 기반 인기글/키워드 조회 및 fallback
   - 관리자용 커뮤니티 집계 상태 점검 API

### 2.2 주요 문제

1. 한글 문자열 깨짐
   - 프론트 상수, 화면 문구, 버튼, placeholder, toast 문구에 깨진 한글이 남아 있다.
   - 백엔드 카테고리/정렬 상수, 상대 시간 문구, 키워드 stopword, migration backfill에도 깨진 값이 남아 있다.
   - 테스트도 깨진 문자열을 기대값으로 사용하는 부분이 있어 현재 테스트만으로는 방지할 수 없다.

2. 권한 정책 불일치
   - 기존 Phase0 문서는 MVP에서 글쓰기/댓글/좋아요를 아파트 인증 회원에게만 허용하는 방향이다.
   - 현재 구현은 전체 커뮤니티 글쓰기는 일반 회원도 가능하고, 댓글/좋아요는 인증 회원만 가능하다.

3. 마이페이지 커뮤니티 조회 보안
   - 내 글/내 댓글 API가 `nickname` 요청 파라미터 기반이다.
   - 커뮤니티 GET API가 공개 허용되어 있어 타인의 닉네임을 알면 조회 가능성이 있다.

4. 검색과 페이징 미흡
   - 게시글 목록은 서버 페이징 없이 전체 리스트를 가져온다.
   - 검색은 프론트 클라이언트 필터만 적용된다.
   - 프론트 서비스에는 `q` 파라미터 준비 코드가 있지만 백엔드 컨트롤러/서비스는 아직 처리하지 않는다.

5. 좋아요/댓글/삭제 후 캐시 갱신 범위 부족
   - 상세 화면 일부만 갱신하고 목록, 사이드바, 마이페이지 카운트가 늦게 맞을 수 있다.

6. 사이드 정보영역 보완 필요
   - UI P0 요구사항은 상당 부분 반영되어 있으나 빈 상태, 에러 처리, 더보기/정렬 전환, 실제 작성시각/최종 댓글시각 메타가 일부 부족하다.

## 3.2 P0: 권한 정책 확정 및 일관화

### 목표

글쓰기, 댓글, 좋아요 권한 정책을 문서와 구현에서 동일하게 만든다.

### 정책 결정 필요

다음 두 안 중 하나를 확정한다.

1. MVP 문서 유지안
   - 게스트: 읽기만 가능
   - 로그인 일반 회원: 읽기만 가능
   - 아파트 인증 회원: 글쓰기, 댓글, 좋아요 가능

2. 전체 커뮤니티 완화안
   - 게스트: 읽기만 가능
   - 로그인 일반 회원: 전체 커뮤니티 글쓰기 가능
   - 아파트 인증 회원: 전체 커뮤니티와 우리 아파트 글쓰기, 댓글, 좋아요 가능

### 권장안

MVP 문서와 기존 인증 UX에 맞춰 1안을 우선 적용한다.

### 대상

1. `frontend/src/router/VerifiedRoute.tsx`
2. `frontend/src/pages/WritePage.tsx`
3. `frontend/src/pages/PostDetailPage.tsx`
4. `backend/src/main/java/com/realestate/service/CommunityService.java`
5. `mdfile/커뮤니티탭/공통게시판_Phase0_정책확정_체크리스트.md`

### 작업 내용

1. 프론트 `/write` 접근 조건을 정책에 맞게 정리
2. 백엔드 `createPost`, `createComment`, `toggleLike` 검증 로직 일관화
3. 제한 시 표시되는 안내 문구 통일
4. 정책 문서와 코드 주석 불일치 제거

### 완료 기준

1. 프론트에서 막는 권한과 백엔드에서 막는 권한이 동일해야 한다.
2. 일반 회원이 제한된 기능을 직접 API 호출해도 서버가 차단해야 한다.
3. 게스트, 일반 회원, 인증 회원별 수동 QA가 완료되어야 한다.

## 3.3 P1: 마이페이지 커뮤니티 조회 보안 보완

### 목표

내 글/내 댓글 조회를 요청 파라미터가 아닌 인증 사용자 기준으로 변경한다.

### 대상

1. `backend/src/main/java/com/realestate/web/controller/CommunityController.java`
2. `backend/src/main/java/com/realestate/service/CommunityService.java`
3. `backend/src/main/java/com/realestate/auth/SecurityConfig.java`
4. `frontend/src/services/communityService.ts`
5. `frontend/src/components/features/mypage/MyPostList.tsx`
6. `frontend/src/components/features/mypage/MyCommentList.tsx`

### 작업 내용

1. `GET /api/community/my/posts`
   - 기존: `?nickname=...`
   - 변경: 인증 principal의 nickname 또는 userId 사용

2. `GET /api/community/my/comments`
   - 기존: `?nickname=...`
   - 변경: 인증 principal의 nickname 또는 userId 사용

3. API 보안 정책 변경
   - `/api/community/my/**`는 authenticated 필요

4. 프론트 요청에서 nickname 파라미터 제거

### 완료 기준

1. 로그인하지 않은 사용자는 내 글/내 댓글 API를 호출할 수 없어야 한다.
2. 다른 사용자의 nickname을 넣어도 조회할 수 없어야 한다.
3. 마이페이지 내 글/내 댓글 UI는 기존처럼 정상 표시되어야 한다.

## 3.4 P1: 게시글 목록 검색 및 페이징 서버화

### 목표

게시글 목록을 대량 데이터에서도 안정적으로 조회하도록 서버 페이징과 서버 검색을 도입한다.

### 대상

1. `backend/src/main/java/com/realestate/web/controller/CommunityController.java`
2. `backend/src/main/java/com/realestate/service/CommunityService.java`
3. `backend/src/main/java/com/realestate/domain/repository/CommunityPostRepository.java`
4. `frontend/src/services/communityService.ts`
5. `frontend/src/pages/CommunityPage.tsx`
6. `frontend/src/utils/communitySearch.ts`

### API 설계

```text
GET /api/community/posts
  ?scope=GLOBAL
  &boardCode=BLAH
  &sortType=LATEST
  &q=검색어
  &page=0
  &size=20
```

### 작업 내용

1. `sortType`을 표시 문자열이 아닌 안정적인 코드로 변경
   - `LATEST`
   - `POPULAR`
   - `COMMENT`

2. 응답을 리스트에서 페이지 응답으로 전환

```json
{
  "content": [],
  "page": 0,
  "size": 20,
  "totalElements": 0,
  "totalPages": 0,
  "hasNext": false
}
```

3. 제목/본문 검색 조건 추가
4. 프론트 무한스크롤 또는 더보기 방식 도입
5. 기존 URL 파라미터와 호환 처리

### 완료 기준

1. 검색어 입력 시 서버 검색 결과가 표시되어야 한다.
2. 게시글이 많아도 최초 목록 조회가 전체 row를 가져오지 않아야 한다.
3. 기존 scope/boardCode 필터와 정렬이 함께 동작해야 한다.

### 목표

댓글, 좋아요, 삭제 후 목록/상세/사이드바/마이페이지 데이터가 일관되게 갱신되도록 한다.

### 대상

1. `frontend/src/pages/PostDetailPage.tsx`
2. `frontend/src/components/features/mypage/MyPostList.tsx`
3. `frontend/src/components/features/mypage/MyCommentList.tsx`
4. `frontend/src/hooks/useSidebarData.ts`
5. `frontend/src/services/communityService.ts`

### 작업 내용

1. query key 상수화 검토
2. 댓글 작성 성공 시 invalidate 대상
   - `['community', 'comments', postId]`
   - `['community', 'post', postId]`
   - `['community', 'posts']`
   - `['community', 'popular']`
   - `['community', 'hotComments']`
   - `['community', 'my-comments']`

3. 댓글 삭제 성공 시 동일 범위 갱신
4. 좋아요 성공 시 목록/상세/인기글 갱신
5. 게시글 삭제 성공 시 목록/상세/사이드바/마이페이지 갱신

### 완료 기준

1. 좋아요 후 상세와 목록의 좋아요 수가 일치해야 한다.
2. 댓글 작성/삭제 후 상세와 목록의 댓글 수가 일치해야 한다.
3. 인기글/댓글 많은 글 순위가 staleTime을 기다리지 않아도 합리적으로 갱신되어야 한다.

2. 로그인 일반 회원
   - 정책에 맞는 글쓰기 제한 확인
   - 댓글/좋아요 제한 확인
   - 마이페이지 내 글/내 댓글 정상 조회

3. 아파트 인증 회원
   - 전체 커뮤니티 글쓰기
   - 우리 아파트 글쓰기
   - 댓글 작성/삭제
   - 좋아요 토글
   - 게시글 삭제

4. 반응형
   - 모바일 390px
   - 태블릿
   - 데스크톱

## 6. 완료 정의

1. 사용자 화면에 깨진 한글이 없어야 한다.
2. 권한 정책이 문서, 프론트, 백엔드에서 동일해야 한다.
3. 마이페이지 커뮤니티 조회가 인증 사용자 기준으로만 동작해야 한다.
4. 커뮤니티 목록은 검색/페이징 기준으로 동작해야 한다.
5. 댓글/좋아요/삭제 후 관련 화면의 수치가 일관되게 갱신되어야 한다.
6. 프론트 빌드와 백엔드 테스트가 모두 통과해야 한다.
7. 주요 커뮤니티 플로우 수동 QA가 완료되어야 한다.
