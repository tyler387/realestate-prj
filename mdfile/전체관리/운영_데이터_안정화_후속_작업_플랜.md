---

## 1. 배경

최신 커밋에서 다음 P0 성격의 안정화가 완료되었다.

1. 관리자 수집 API에 `X-Admin-Api-Key` 인증 추가
2. 장시간 수집 작업에 단일 job lock/status API 추가
3. 실거래 상세 응답에 `displayedCount`, `limit`, `hasMore` 추가
4. 실거래 차트와 거래내역의 과도한 렌더링/빈 상태 UX 보강
5. 백엔드 테스트와 프론트 빌드 통과

다음 단계에서는 기능을 급하게 넓히기보다, 운영 중 데이터가 어떻게 들어오고 어디서 누락되는지 확인 가능한 구조를 먼저 만든다.

## 2. 전체 우선순위

| 우선순위 | 작업 | 목적 |
|---|---|---|
| P1 | 백엔드 전세/월세 요청 방어 정책 고정 | 프론트 우회 요청으로 빈 결과 UX가 재발하지 않게 함 |

## P1. 백엔드 전세/월세 요청 방어 정책 고정

### 문제

프론트는 현재 `JEONSE`, `MONTHLY`를 매매 기준으로 보정한다.

```text
frontend/src/utils/tradeType.ts
frontend/src/hooks/useApartmentTrade.ts
frontend/src/stores/tradeFilterStore.ts
```

하지만 백엔드 API는 직접 요청으로 `dealType=JEONSE` 또는 `dealType=MONTHLY`가 들어오면 `LEASE`, `MONTHLY`로 해석할 수 있다.

```text
backend/src/main/java/com/realestate/service/ApartmentService.java
backend/src/main/java/com/realestate/service/TradeStatsService.java
```

현재 수집기는 매매 API만 사용하므로, 프론트를 우회한 요청에서 빈 결과가 다시 발생할 수 있다.

### 결정할 정책

권장 정책:

```text
JEONSE  -> SALE
LEASE   -> SALE
MONTHLY -> SALE
```

이유:

1. P0에서 이미 "전세/월세는 준비 중이며 현재는 매매 기준으로 조회"라는 UX를 채택했다.
2. 프론트와 백엔드 정책이 같아진다.
3. 저장 필터, URL query, 직접 API 호출이 모두 같은 결과를 만든다.

대안:

```text
JEONSE/MONTHLY 요청 시 400 Bad Request
```

단, 이 방식은 프론트 우회 요청에는 명확하지만, 저장 필터/과거 URL 복구 UX에는 더 거칠다.

### 작업 파일

```text
backend/src/main/java/com/realestate/service/ApartmentService.java
backend/src/main/java/com/realestate/service/TradeStatsService.java
backend/src/test/java/com/realestate/service/ApartmentServiceTradeRecordTest.java
```

필요 시 추가:

```text
backend/src/test/java/com/realestate/service/TradeStatsServiceTest.java
```

### 구현 작업

1. `ApartmentService.resolveTradeType()`에서 `JEONSE`, `LEASE`, `MONTHLY`를 `SALE`로 보정한다.
2. `TradeStatsService.resolveTradeType()`도 동일하게 보정한다.
3. 보정 정책을 메서드명이나 주석으로 명확히 남긴다.
4. 테스트에 아래 케이스를 추가한다.

```text
dealType=SALE    -> SALE 조건으로 조회
dealType=JEONSE  -> SALE 조건으로 조회
dealType=LEASE   -> SALE 조건으로 조회
dealType=MONTHLY -> SALE 조건으로 조회
dealType=null    -> 상세 API는 전체 또는 기존 정책 유지 여부 확인
```

### 완료 기준

1. 백엔드로 직접 `dealType=JEONSE`, `dealType=MONTHLY`를 보내도 전세/월세 빈 결과가 나오지 않는다.
2. 상세 거래내역, 가격 히스토리, 랭킹 API의 정책이 일관된다.
3. 백엔드 테스트가 통과한다.

### 검증

```bash
cd backend
.\gradlew.bat test
```

수동 확인:

```text
GET /api/v1/apartments/{id}/trades?dealType=JEONSE
GET /api/v1/apartments/{id}/price-history?dealType=MONTHLY
GET /api/v1/trades/top-apartments?dealType=JEONSE
```

## P2. 관리자 수집 Job 운영성 보강

### 문제

현재 관리자 수집 job은 단일 lock과 최근 job 상태를 인메모리로 제공한다.

```text
backend/src/main/java/com/realestate/service/AdminCollectionJobService.java
backend/src/main/java/com/realestate/web/controller/AdminCollectController.java
```

운영 관점에서 아직 부족한 점:

1. 서버 재시작 시 job 이력이 사라진다.
2. `CompletableFuture.runAsync()`가 기본 풀을 사용한다.
3. 진행률이 없어 장시간 수집 중 어디까지 진행됐는지 알기 어렵다.
4. 실패 원인이 짧은 메시지만 남는다.

### 작업 파일

```text
backend/src/main/java/com/realestate/service/AdminCollectionJobService.java
backend/src/main/java/com/realestate/web/controller/AdminCollectController.java
backend/src/main/java/com/realestate/collect/RealTradeCollector.java
backend/src/main/java/com/realestate/collect/ApartmentComplexCollector.java
backend/src/main/resources/db/migration/
```

신규 후보:

```text
backend/src/main/java/com/realestate/domain/entity/CollectionJob.java
backend/src/main/java/com/realestate/domain/repository/CollectionJobRepository.java
backend/src/main/java/com/realestate/config/CollectionJobExecutorConfig.java
backend/src/main/java/com/realestate/web/dto/AdminJobProgressDto.java
```

### 구현 작업

1. 전용 executor 추가

```text
ThreadPoolTaskExecutor
corePoolSize = 1
maxPoolSize = 1
queueCapacity = 0 또는 1
threadNamePrefix = collection-job-
```

2. job 상태를 DB에 저장

```text
job_id
type
status: RUNNING/SUCCEEDED/FAILED/CANCELLED
started_at
finished_at
error_message
progress_message
total_count
processed_count
saved_count
skipped_count
duplicate_count
```

3. 수집기에서 진행 상황 callback을 받을 수 있게 구조 변경

```text
collectAllDistricts(months, progressCallback)
collectDistrict(lawdCd, months, progressCallback)
collectComplexes(sigunguCd, progressCallback)
```

4. 관리자 API 응답 확장

```text
GET /api/v1/admin/jobs
GET /api/v1/admin/jobs/current
GET /api/v1/admin/jobs/{jobId}
```

5. 실패 시 stack trace 전체는 로그에 남기고, API 응답에는 짧은 실패 메시지만 제공한다.

### 완료 기준

1. 서버 재시작 후에도 최근 job 이력을 조회할 수 있다.
2. 동시에 두 개의 수집 job이 실행되지 않는다.
3. 진행 중인 job에서 현재 구/월/page 또는 처리 건수를 확인할 수 있다.
4. 실패 job의 원인을 운영자가 추적할 수 있다.

### 검증

```bash
cd backend
.\gradlew.bat test
```

수동 확인:

```text
POST /api/v1/admin/collect?months=1
GET  /api/v1/admin/jobs/current
GET  /api/v1/admin/jobs
```

동시 실행 확인:

```text
첫 번째 수집 job 실행 중 두 번째 POST 요청은 409 Conflict 반환
```

## P3. 수집 데이터 품질 리포트

### 문제

현재 수집 중 아파트 매칭 실패는 카운트로만 남는다.

```text
skippedNoApartment
duplicateTrades
savedTrades
```

하지만 어떤 단지가 실패했는지 알 수 없으면 수집 품질을 개선하기 어렵다.

### 작업 파일

```text
backend/src/main/java/com/realestate/collect/RealTradeCollector.java
backend/src/main/java/com/realestate/collect/ApartmentComplexCollector.java
backend/src/main/resources/db/migration/
```

신규 후보:

```text
backend/src/main/java/com/realestate/domain/entity/CollectionIssue.java
backend/src/main/java/com/realestate/domain/repository/CollectionIssueRepository.java
backend/src/main/java/com/realestate/web/controller/AdminCollectionIssueController.java
backend/src/main/java/com/realestate/web/dto/CollectionIssueDto.java
```

### DB 설계 후보

```sql
CREATE TABLE collection_issue (
    id BIGSERIAL PRIMARY KEY,
    job_id VARCHAR(64),
    source_type VARCHAR(50) NOT NULL,
    issue_type VARCHAR(50) NOT NULL,
    sigungu VARCHAR(50),
    eup_myeon_dong VARCHAR(50),
    apt_name VARCHAR(255),
    lawd_cd VARCHAR(20),
    deal_ymd VARCHAR(6),
    raw_payload JSONB,
    message TEXT,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

### issue_type 후보

```text
NO_APARTMENT_MATCH
INVALID_TRADE_DATE
INVALID_AMOUNT
API_ERROR
NO_COORDINATE
INVALID_COMPLEX_PAYLOAD
```

### 구현 작업

1. 거래 수집 중 아파트 매칭 실패를 `collection_issue`에 저장한다.
2. 좌표 조회 실패, API 오류도 동일하게 저장한다.
3. 관리자 조회 API를 추가한다.

```text
GET /api/v1/admin/collection-issues
GET /api/v1/admin/collection-issues?issueType=NO_APARTMENT_MATCH
GET /api/v1/admin/collection-issues?jobId={jobId}
```

4. 같은 단지 실패가 과도하게 중복 저장되지 않도록 중복 기준을 정한다.

권장 중복 기준:

```text
source_type + issue_type + sigungu + eup_myeon_dong + apt_name + deal_ymd
```

### 완료 기준

1. 수집 실패가 단순 로그가 아니라 DB에 남는다.
2. 운영자가 어떤 단지/지역에서 누락이 많은지 확인할 수 있다.
3. 향후 수동 매칭 또는 매칭 룰 개선의 입력 데이터로 사용할 수 있다.

### 검증

```bash
cd backend
.\gradlew.bat test
```

수동 확인:

```text
매칭 실패가 발생하는 샘플 데이터를 넣고 collection_issue 저장 여부 확인
관리자 API에서 issue 목록 조회
```

## P4. 실거래 필터 로직 공통화

### 문제

현재 상세 API와 랭킹 API에서 기간/가격/면적/층/연식/거래유형 필터 해석 로직이 중복된다.

```text
backend/src/main/java/com/realestate/service/ApartmentService.java
backend/src/main/java/com/realestate/service/TradeStatsService.java
```

중복이 유지되면 이후 전세/월세, 지역 필터, 저장 필터를 확장할 때 API별 결과가 달라질 수 있다.

### 작업 파일

```text
backend/src/main/java/com/realestate/service/ApartmentService.java
backend/src/main/java/com/realestate/service/TradeStatsService.java
```

신규 후보:

```text
backend/src/main/java/com/realestate/service/trade/TradeFilterCriteria.java
backend/src/main/java/com/realestate/service/trade/TradeFilterCriteriaResolver.java
backend/src/test/java/com/realestate/service/trade/TradeFilterCriteriaResolverTest.java
```

### 구현 작업

1. 공통 criteria record를 만든다.

```text
startDate
endDate
tradeType
minPrice
maxPrice
minArea
maxArea
minFloor
maxFloor
minAge
maxAge
onlyNew
onlyLarge
onlyHot
complexKeyword
excludeOutliers
```

2. `ApartmentService`와 `TradeStatsService`에서 공통 resolver를 사용한다.
3. API별로 다른 정책이 필요한 부분은 명시적으로 옵션을 둔다.

4. resolver 단위 테스트를 먼저 작성한다.

### 완료 기준

3. 전세/월세 정책 변경 시 수정 지점이 줄어든다.

## P5. 전세/월세 실제 수집 기반 설계 및 구현

### 전제

P5는 바로 구현하지 않는다. 아래 작업이 먼저 끝난 뒤 진행한다.

1. P1 백엔드 unsupported rent 정책 고정
2. P2 관리자 job 운영성 보강
3. P3 수집 품질 리포트
4. P4 필터 로직 공통화

### 문제

현재 수집기는 매매 API만 사용한다.

```text
getRTMSDataSvcAptTrade
```

전세/월세는 별도 API와 별도 금액 모델이 필요하다.

```text
getRTMSDataSvcAptRent
```

### 설계 결정 사항

1. DB 스키마

후보 A: 기존 `real_trade` 확장

```text
trade_type: SALE/LEASE/MONTHLY
trade_amount: 매매가 또는 전세 보증금
deposit_amount: 전세/월세 보증금
monthly_rent_amount: 월세
```

후보 B: 매매와 임대 테이블 분리

```text
real_trade
real_rent
```

권장:

```text
초기에는 기존 real_trade 확장
```

이유:

1. 상세/차트/랭킹 API를 재사용하기 쉽다.
2. 기존 `trade_type` 구조와 맞다.
3. 월세 계산 기준만 별도로 분기하면 된다.

2. 월세 차트 기준

결정 필요:

```text
월세 보증금 기준
월세 월 납입액 기준
전월세 환산가 기준
보증금/월세를 별도 축으로 표시
```

권장:

```text
P5 1차: 전세는 보증금 기준, 월세는 목록만 제공하고 차트는 보류
P5 2차: 월세 전용 차트 설계
```

3. UI 정책

전세/월세 데이터가 실제 들어오기 전까지:

```text
준비 중 badge 유지
```

전세 데이터가 들어온 뒤:

```text
전세 활성화
월세는 준비 중 유지 가능
```

### 작업 파일

```text
backend/src/main/java/com/realestate/collect/RealTradeCollector.java
backend/src/main/java/com/realestate/domain/entity/RealTrade.java
backend/src/main/resources/db/migration/
backend/src/main/java/com/realestate/domain/repository/RealTradeRepository.java
backend/src/main/java/com/realestate/service/ApartmentService.java
frontend/src/utils/tradeType.ts
frontend/src/components/features/trade/TradeTypeFilter.tsx
frontend/src/components/features/trade-sidebar/QuickFilters.tsx
frontend/src/components/features/trade/PriceChart.tsx
frontend/src/components/features/trade/TradeHistoryList.tsx
```

### 구현 순서

1. 임대 API 응답 샘플 수집
2. DB 스키마 확장 migration 작성
3. `RealTrade` 엔티티 확장
4. 임대 수집 메서드 추가
5. 중복 키 정책 확정
6. 전세 데이터만 먼저 API 응답에 포함
7. 프론트에서 전세 필터 활성화
8. 월세는 별도 UX 설계 후 진행

### 완료 기준

1. 전세 데이터가 수집되고 상세 거래내역에서 조회된다.
2. 매매 기존 기능이 깨지지 않는다.
3. 전세/월세가 없는 지역의 빈 상태 UX가 명확하다.
4. 월세 금액 기준이 정해지기 전에는 차트가 잘못된 의미를 주지 않는다.
