# 공통게시판 + 아파트 인증 배지 커뮤니티 확장 플랜
## v1.0 | 블라인드형 전체 커뮤니티와 아파트별 커뮤니티 병행 구조

> 기준일: 2026-05-22  
> 대상: HomeBlind Frontend(커뮤니티 탭/글쓰기/상세/마이페이지) / Backend(게시글 도메인/API/집계)  
> 목적: 호갱노노식 아파트별 커뮤니티에 블라인드식 공통 게시판을 결합해, 아파트 소속 신호가 드러나는 계급형 부동산 커뮤니티 구조를 만든다.

---

## 1. 배경

현재 커뮤니티는 `aptId`가 필수인 아파트별 게시판 중심이다. 이 구조는 같은 단지 주민 간 대화에는 적합하지만, 블라인드의 `블라블라`, `주식`, `부동산`, `연애`처럼 넓은 주제의 전체 커뮤니티를 만들기 어렵다.

목표 서비스는 단지 인증을 커뮤니티 정체성으로 삼되, 대화 공간은 특정 아파트에만 갇히지 않게 확장해야 한다. 공통 게시판에서도 인증 사용자는 닉네임 옆에 `아파트 인증 배지(아파트명)`가 노출되어야 한다. 이 배지는 신뢰 신호이자 소속/위상 신호로 작동한다.

---

## 2. 핵심 방향

1. 커뮤니티를 `아파트별 게시판`과 `공통 게시판`으로 분리한다.
2. 게시글은 어느 게시판에 속하는지 나타내는 `boardScope`를 가진다.
3. 공통 게시판 글은 특정 `aptId`에 종속되지 않을 수 있다.
4. 작성자 인증 정보는 게시판 소속과 별개로 보존한다.
5. 인증 계정은 공통 게시판에서도 `아파트명 인증` 배지를 노출한다.
6. 미인증/비회원은 읽기 중심으로 두고, 쓰기/댓글/좋아요는 인증 단계에 따라 제한한다.

## 3. 게시판 IA

## 3.1 상위 탭

1. `전체 커뮤니티`
2. `우리 아파트`

### 전체 커뮤니티

블라인드식 공통 주제 게시판이다. `aptId` 없이 조회 가능해야 한다.

초기 카테고리:

1. `블라블라`
2. `부동산`
3. `주식`
4. `연애`
5. `이직/직장`
6. `질문`

### 우리 아파트

기존 호갱노노식 아파트별 커뮤니티다. 기존 `aptId` 기반 목록/인기글/키워드/사이드 정보 영역을 유지한다.

초기 카테고리:

1. `전체`
2. `자유`
3. `질문`
4. `정보`
5. `실거래`
6. `민원/하자`

## 4. 사용자 표시 정책

## 4.1 작성자 표시

공통 게시판과 아파트별 게시판 모두 작성자 영역은 아래 구조로 통일한다.

```text
닉네임
[아파트 인증: 래미안 원베일리] [회원 등급/활동 배지]
```

### 인증 상태별 표시

1. 아파트 인증 완료: `아파트 인증: {아파트명}`
2. 회원가입만 완료: `미인증 회원`
3. 게스트: 작성 불가, 읽기 가능

## 4.2 개인정보/안전 정책

1. 동/호수는 절대 노출하지 않는다.
2. 아파트명 배지는 사용자가 숨길 수 없게 하는 것을 기본 정책으로 한다.
3. 다만 운영 리스크가 큰 경우 P1에서 `아파트명 일부 마스킹` 정책을 검토한다.
4. 게시글 작성 시점의 인증 아파트명을 스냅샷으로 저장한다.
5. 사용자가 이사/재인증해도 기존 글의 작성 당시 배지는 유지한다.

## 5. 데이터 모델 설계

## 5.1 현재 제약

현재 `posts` 테이블과 `CommunityPost` 엔티티는 `apt_id nullable=false` 구조다. 따라서 공통 게시판 글을 저장하려면 아래 중 하나가 필요하다.

권장안: 단일 posts 테이블 확장

1. `apt_id` nullable 허용
2. `board_scope` 추가: `GLOBAL`, `APARTMENT`
3. `board_code` 추가: `BLAH`, `REAL_ESTATE`, `STOCK`, `DATING`, `APT_FREE` 등
4. 작성자 인증 스냅샷 필드 추가

대안: global_posts 별도 테이블

1. 빠른 분리는 쉽지만 댓글/좋아요/집계/마이페이지 중복 구현이 커진다.
2. 현 프로젝트 규모에서는 권장하지 않는다.

## 5.2 posts 테이블 확장안

추가/변경 컬럼:

```sql
ALTER TABLE posts ALTER COLUMN apt_id DROP NOT NULL;
ALTER TABLE posts ADD COLUMN board_scope VARCHAR(20) NOT NULL DEFAULT 'APARTMENT';
ALTER TABLE posts ADD COLUMN board_code VARCHAR(40) NOT NULL DEFAULT 'APT_FREE';
ALTER TABLE posts ADD COLUMN author_user_id BIGINT;
ALTER TABLE posts ADD COLUMN author_verified_apt_id BIGINT;
ALTER TABLE posts ADD COLUMN author_verified_apt_name VARCHAR(100);
ALTER TABLE posts ADD COLUMN author_verification_label VARCHAR(120);
```

### 필드 의미

1. `apt_id`: 아파트별 게시판 대상 단지. 공통 게시판에서는 null.
2. `board_scope`: 게시판 범위. `GLOBAL` 또는 `APARTMENT`.
3. `board_code`: 실제 게시판/카테고리 코드.
4. `author_verified_apt_id`: 작성자의 인증 단지 id 스냅샷.
5. `author_verified_apt_name`: 작성자의 인증 단지명 스냅샷.
6. `author_verification_label`: UI 표시용 스냅샷. 예: `아파트 인증: 래미안 원베일리`.

## 5.3 댓글 표시 확장

현재 댓글은 `author_apt_id`, `author_apt_name` 필드가 이미 있는 구조다. 공통 게시판 댓글에도 동일 필드를 활용한다.

보강 필요:

1. 댓글 작성 시 로그인 사용자의 현재 인증 아파트를 서버에서 스냅샷으로 채운다.
2. 프론트가 임의로 `authorAptName`을 보내는 구조는 장기적으로 제거한다.

## 6. API 설계

## 6.1 게시글 목록

현재:

```text
GET /api/community/posts?aptId=...
```

변경:

```text
GET /api/community/posts?scope=GLOBAL&boardCode=BLAH&sortType=최신순
GET /api/community/posts?scope=APARTMENT&aptId=3100&boardCode=APT_FREE&sortType=최신순
```

### 요청 파라미터

1. `scope`: `GLOBAL` 또는 `APARTMENT`
2. `aptId`: `APARTMENT`일 때 필수
3. `boardCode`: 선택. 없으면 해당 scope의 전체
4. `sortType`: `최신순`, `인기순`, `댓글순`

## 6.2 게시글 생성

현재:

```json
{
  "aptId": 3100,
  "category": "자유",
  "title": "...",
  "content": "...",
  "authorNickname": "...",
  "complexName": "..."
}
```

변경:

```json
{
  "scope": "GLOBAL",
  "boardCode": "BLAH",
  "title": "...",
  "content": "..."
}
```

서버가 인증 토큰에서 작성자를 식별하고 아래 정보를 채운다.

1. `authorNickname`
2. `authorUserId`
3. `authorVerifiedAptId`
4. `authorVerifiedAptName`
5. `authorVerificationLabel`

## 6.3 인기글/키워드

공통 게시판과 아파트별 게시판을 분리 집계한다.

```text
GET /api/community/posts/popular?scope=GLOBAL&boardCode=BLAH
GET /api/community/posts/hot-comments?scope=GLOBAL&boardCode=BLAH
GET /api/community/keywords?scope=GLOBAL&boardCode=BLAH
```

기존 아파트별 사이드바는 다음처럼 유지한다.

```text
GET /api/community/posts/popular?scope=APARTMENT&aptId=3100
GET /api/community/keywords?scope=APARTMENT&aptId=3100
```

공통 게시판 글 카드:

```text
제목
본문 미리보기
닉네임 · 아파트 인증: 래미안 원베일리 · 12분 전
```

아파트별 게시판 글 카드:

## 7.3 글쓰기 진입

글쓰기 화면 진입 시 현재 선택된 scope와 boardCode를 유지한다.

1. 전체 커뮤니티 > 주식에서 글쓰기: `scope=GLOBAL`, `boardCode=STOCK`
2. 우리 아파트 > 실거래에서 글쓰기: `scope=APARTMENT`, `aptId=current`, `boardCode=APT_TRADE`

인증 상태별 정책:

1. 게스트: 로그인 바텀시트
2. 회원 미인증: 읽기 가능, 글쓰기 시 아파트 인증 유도
3. 아파트 인증 회원: 글쓰기/댓글/좋아요 가능

1. Flyway migration 추가
2. `CommunityPost` 엔티티 확장

## 11. QA 체크리스트

1. 공통 게시판은 `aptId` 없이 목록 조회가 된다.
2. 우리 아파트 게시판은 기존처럼 선택 아파트 기준으로만 보인다.
3. 공통 게시판 글에도 인증 사용자의 아파트명 배지가 표시된다.
4. 미인증 회원은 글쓰기/댓글 작성 시 인증 안내를 본다.
5. 기존 아파트별 게시글이 migration 후 정상 조회된다.
6. 인기글/댓글많은글/키워드가 scope를 섞지 않는다.
7. 마이페이지 내 글/댓글 목록에서 공통 게시판 글도 보인다.
8. 게시글 상세 진입, 좋아요, 댓글 작성/삭제가 기존과 동일하게 동작한다.

## 12. 리스크

1. `aptId nullable` 변경으로 기존 조회 쿼리가 null을 처리하지 못할 수 있다.
   - 대응: repository 메서드를 scope별로 명확히 분리한다.

2. 인증 배지가 과도한 개인정보 노출로 느껴질 수 있다.
   - 대응: 동/호수는 절대 노출하지 않고 아파트명만 노출한다.

3. 공통 게시판 도입으로 기존 아파트별 게시판의 트래픽이 분산될 수 있다.
   - 대응: 상단 IA에서 `우리 아파트`를 명확히 유지하고, 공통 게시판 글 카드에도 아파트 배지를 강하게 노출한다.

4. 기존 `category`와 신규 `boardCode`가 혼재될 수 있다.
   - 대응: migration 후 서버 응답은 `boardCode` 중심으로 통일하고, 프론트 표시명은 매핑 테이블로 관리한다.

## 13. 추천 구현 순서

1. DB migration과 도메인 모델 확장
2. 기존 아파트별 게시판 회귀 테스트
3. 공통 게시판 목록 조회 API 추가
4. 공통 게시판 프론트 탭 추가
5. 공통 게시판 글쓰기/상세/댓글 연결
6. 인증 배지 UI 전역 적용
7. 공통 게시판 인기/키워드 집계 확장

---

## 14. 1차 완료 기준

1. `전체 커뮤니티 > 블라블라`에서 `aptId` 없이 게시글 목록을 볼 수 있다.
2. 인증 회원은 공통 게시판에 글을 작성할 수 있다.
3. 작성자 영역에 `아파트 인증: {아파트명}`이 표시된다.
4. 기존 `우리 아파트` 게시판은 기존 기능이 깨지지 않는다.
5. 공통 게시판과 아파트별 게시판의 목록/상세 URL 상태가 분리된다.
